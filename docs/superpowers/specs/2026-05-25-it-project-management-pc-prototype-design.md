# IT项目管理系统 PC网页原型设计

日期：2026-05-25

## 目标

制作一个可展示的 PC 网页端原型，用于呈现 IT 项目管理系统的核心产品形态。原型不接真实后端，不要求各系统真实联通，但需要通过静态模拟数据、页面切换、弹窗、筛选、状态标签和详情面板，展示 PRD 中的主要角色、流程和管理视图。

首版只覆盖 PC 网页端。移动端、企业微信内嵌应用、MCP 接口和机器人能力在界面中作为入口或集成状态展示，不做真实功能。

## 目标用户与演示重点

原型覆盖以下角色视角：

- 高管：查看全局项目进度、资源投入、预算使用和交付评分。
- 项目经理：管理项目规则、资源、预算、风险、绩效和权限。
- 产品经理：承接需求，分析实现方式，推进项目和验收。
- 部门负责人身份：不是独立角色，在原业务角色基础上查看部门需求，调整优先级，监控投入。
- 开发：查看任务、维护状态、填报工时。

演示重点是“企业级 IT 项目全生命周期管理”：从需求提出，到分析、立项、开发、测试、验收、评价，再到高管报表和权限治理。

## 产品形态

采用现代企业级后台布局：

- 左侧深色主导航，承载主要模块。
- 顶部栏提供全局搜索、角色视角切换、通知入口和用户信息。
- 主内容区以数据表格、指标卡、看板、详情抽屉、表单弹窗和图表为主。
- 视觉风格克制、清晰、偏办公场景，避免营销化大图和装饰性卡片堆叠。

顶部角色视角切换用于演示不同用户的关注重点。切换角色后，不实现完整权限系统，但会调整首页核心指标、快捷入口和部分文案，使演示更贴近真实多角色系统。

## 页面范围

### 1. 项目驾驶舱

作为首屏页面，展示系统全局状态：

- 项目总数、进行中项目、逾期风险、预算使用率、人天投入、平均交付评分。
- 项目阶段分布，包括待评审、开发中、测试中、待验收、已上线。
- 资源投入趋势和预算消耗趋势。
- 风险项目列表，展示项目负责人、风险等级、风险原因和应对状态。
- 待办事项，包括需求评审、资源审批、验收提醒、逾期任务。

角色切换时，驾驶舱重点变化：

- 高管看整体项目、预算和绩效。
- 项目经理看风险、资源、预算和审批。
- 产品经理看待分析需求、迭代进度和验收。
- 运营叠加运营中心负责人身份后，看部门需求和资源投入。
- 开发看个人任务和工时。

### 2. 需求管理

展示需求从提交到验收的管理过程：

- 需求列表：需求编号、名称、提出人、团队、优先级、状态、当前处理人、进度、期望上线时间。
- 筛选项：状态、优先级、团队、处理人、实现方式。
- 新建需求弹窗：需求名称、业务目标、描述、优先级、附件上传区域。
- 需求详情抽屉：基础信息、处理进度、里程碑、评论记录、关联项目、验收评分。
- 需求优先级调整：通过下拉或快捷按钮展示部门负责人调整能力。

需求状态包括：待评审、分析中、开发中、测试中、待验收、已完成、暂停。

### 3. 项目管理

展示需求转项目后的项目计划和推进：

- 项目列表：项目名称、关联需求、负责人、实现方式、当前阶段、进度、预算、人天、风险状态。
- 项目详情：阶段计划、里程碑、关联任务、资源分配、预算使用、风险记录。
- 甘特式阶段条：用横向阶段进度展示评审、设计、开发、测试、验收、上线。
- 实现方式标识：内部实现或外部供应商。

项目详情不做复杂编辑器，只需要足够支持展示项目经理和产品经理如何跟踪项目。

### 4. 任务与工时

展示开发角色的任务承接和工时填报：

- 看板列：待开始、进行中、测试中、已完成、暂停。
- 任务卡片：任务名称、所属项目、负责人、预计工时、实际工时、截止日期。
- 任务详情弹窗：任务描述、排期、状态、工时填报记录、备注。
- 快捷状态变更：模拟把任务从一个状态切换到另一个状态。

状态变化只在前端临时生效，不要求持久化到后端。

### 5. 资源与预算

展示项目经理对资源池和成本的掌控：

- 人员资源池：姓名、角色、技能栈、可用工时、已分配工时、负载状态。
- 项目投入视图：内部人天投入、外部供应商金额、预算使用率。
- 资源申请列表：产品经理提交的资源申请，展示审批状态。
- 供应商预算条目：供应商名称、项目、合同金额、已使用金额、付款状态。

### 6. 绩效与报表

展示高管和项目经理关注的数据分析：

- 项目交付评分趋势。
- 团队需求完成率。
- 预算偏差排行。
- 项目延期率和风险分布。
- 内部团队、外部供应商、产品经理和项目的评分概览。

图表使用前端静态数据渲染，重点是信息结构和视觉表达。

### 7. 权限管理

展示角色与权限分离体系：

- 用户列表：姓名、部门、角色、状态。
- 角色列表：运营、产品经理、项目经理、开发、高管、管理员；部门负责人作为组织身份展示。
- 权限矩阵：模块权限和数据范围权限，展示查看、新建、编辑、审批、导出等操作粒度。
- 项目维度权限：展示用户可以访问哪些项目或团队数据。

权限管理位于管理员的系统设置下，只做展示和表单交互，不做真实鉴权。

### 8. 消息通知

展示系统通知能力：

- 通知列表：需求状态变更、任务分配、里程碑达成、验收提醒、逾期预警。
- 通知渠道标识：站内信、企业微信。
- 订阅设置：按通知类型开关订阅。

企业微信通知作为渠道标签展示，不做真实集成。

## 交互范围

原型需要支持以下交互：

- 左侧导航切换页面。
- 顶部角色视角切换。
- 表格筛选、搜索和状态标签展示。
- 点击列表项打开详情抽屉或弹窗。
- 新建需求弹窗。
- 任务状态快捷切换。
- 工时填报模拟。
- 通知已读状态切换。

不实现以下内容：

- 真实登录。
- 真实接口请求。
- 真实数据库。
- 企业微信真实授权。
- MCP 接口真实调用。
- 完整权限鉴权。

## 数据设计

使用内置模拟数据，分为以下静态数据集合：

- roles：角色及其首页配置。
- metrics：各角色驾驶舱指标。
- demands：需求列表和详情。
- projects：项目列表、里程碑、预算和风险。
- tasks：任务看板和工时记录。
- resources：人员资源池和资源申请。
- reports：报表图表数据。
- users：用户、角色和权限矩阵。
- notifications：站内信和企业微信通知。

所有数据保存在前端代码中。允许少量交互修改浏览器内存状态，例如任务状态切换和通知已读。

## 技术方案

在当前空目录中创建一个前端单页应用。优先采用 Vite + React + TypeScript，以便快速搭建、组件化维护和本地演示。

主要结构：

- App：管理当前导航模块、角色视角和全局布局。
- data：存放模拟数据。
- components/layout：侧边栏、顶栏、页面容器。
- components/ui：指标卡、状态标签、表格、弹窗、抽屉、图表容器。
- pages：驾驶舱、需求管理、项目管理、任务工时、资源预算、绩效报表、权限管理、消息通知。

若依赖安装受限，可以退化为纯 HTML/CSS/JavaScript 静态原型，但首选 React 方案，因为页面数量多，组件复用能降低实现复杂度。

## 视觉设计要求

- 页面以高信息密度、清晰层级和易扫描为优先。
- 色彩使用中性背景、深色导航、蓝色作为主操作色，并用绿色、橙色、红色表达完成、预警和风险。
- 卡片圆角不超过 8px。
- 工具按钮优先使用图标加 tooltip 或简短文本。
- 表格、看板和报表需要稳定尺寸，避免内容切换时布局跳动。
- 移动端不在首版范围内，但 PC 页面应至少适配常见桌面宽度。

## 验证计划

- 启动本地开发服务器，确认首页能加载。
- 检查所有导航页面可访问。
- 检查角色切换会改变驾驶舱关注内容。
- 检查需求详情、新建需求、任务详情、工时填报、通知订阅等弹窗或抽屉可打开关闭。
- 检查浏览器控制台无明显运行错误。
- 截图检查桌面视口下无明显文字重叠、布局破损或空白主区域。
