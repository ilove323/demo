# IT项目管理系统原型总览与工作记录

最后更新：2026-05-27

> 后续任何原型制作开始前，先阅读本文件和《IT项目管理系统原型演进故事.md》；如果修改改变了稳定业务口径，必须同步更新相关文档。

## 1. 文档目的

本文档用于保存 IT 项目管理系统 H5 响应式原型的长期上下文，避免对话压缩或多人接手后遗失最初需求、后续追加口径、已实现内容和未完成范围。

本文档是原型工作的事实来源之一，重点记录相对稳定的信息：原始 PRD 摘要、追加需求、当前实现范围、mock 数据口径、角色权限口径、PRD 覆盖情况和后续工作规则。

后续新增或调整原型能力时，应按变化性质同步更新：

- 本文档：当需求口径、模块范围、角色权限、mock 数据主线发生变化时更新。
- 《IT项目管理系统原型演进故事.md》：记录会影响后续判断的演进故事、关键口径和设计原因，不做逐项施工日志。

## 2. 原始需求来源

最新 PRD 文件：

`demo/docs/IT项目管理系统需求文档V1.3.docx`

PRD 标题：

`IT项目管理系统产品需求文档 (PRD)`

原始目标：

- 提升 IT 项目全生命周期管理效率。
- 增强项目信息透明度。
- 优化多角色协作流程，减少沟通成本。
- 实现资源管理和预算控制。
- 提供数据支持，辅助高层决策。

V1.3 系统范围：

- PC 端：完整管理界面，适合日常办公和复杂操作。
- 企业微信内嵌 H5：同一套 H5 页面在企业微信内嵌浏览器中自适应访问，提供轻量查看、通知、状态更新、工时填报、需求提交和评分演示。
- 消息通知：站内信、企业微信、订阅配置。
- MCP 接口：面向外部系统和智能机器人提供标准化能力。
- 权限管理：角色与权限分离，支持模块、数据范围、项目维度的权限控制。
- 项目立项 AI 评分：从业务价值、紧急程度、可行性给出加权评分、推荐结论和理由。

当前原型范围：

- 单套 H5 响应式网页端。
- 静态前端原型，不接后端、不接数据库、不做真实登录、不做真实鉴权、不做真实企业微信、真实 AI 或 MCP 调用。

## 3. 原始 PRD 核心要求摘要

### 3.1 角色

- 需求方：提交业务需求、确认产品方案、发起项目申请、查看进度、验收后评分。
- 需求方负责人：业务部门负责人身份，可查看全部需求-项目情况、调整需求重要级别、监控部门投入和验收结果。
- 产品经理：承接需求、需求评审、补充价值评分、迭代版本、资源方案、预算测算、实现决策和验收组织。
- 项目经理 / IT负责人：系统内唯一项目经理，负责项目启动、资源池维护、开发和供应商指派、风险预算治理、项目验收提交。
- 高管：只读查看全局项目、资源预算投入、交付评分和统计分析。
- 开发：查看参与项目，拆解子任务、评估排期、维护任务状态、填报工时。
- 管理员：查看全量数据，维护用户、角色、部门、菜单和系统集成配置。

### 3.2 功能模块

- 需求管理：需求提交、优先级、状态进度、里程碑、验收评分。
- 项目管理：项目计划、阶段、里程碑、资源、预算、风险、项目关系。
- 任务与工时：开发任务、看板状态、工时填报。
- 资源与预算：人员资源池、技能、可用工时、项目预算、内部人天、外部采购金额。
- 绩效与报表：交付评分、完成率、预算偏差、延期率、风险分布、团队和供应商绩效。
- 权限管理：用户、角色、模块权限、数据范围、项目维度权限。
- 消息通知：需求变更、任务分配、里程碑、验收提醒、逾期预警、订阅设置。
- MCP 与企业微信机器人：查询项目、更新任务、提交简易需求、工时填报等能力展示。

## 4. 用户后续追加需求

以下内容来自原始 PRD 之后的追加口径，后续原型必须继续遵守。

### 4.1 单套 H5 响应式网页端

用户最初明确先做 PC 网页端原型；后续确认删除独立移动端预览模式，改为单套 H5 响应式网页。V1.3 中的企业微信移动办公要求在本原型中解释为“企业微信内嵌 H5 自适应访问”。当前仍不做真实企业微信小程序工程、不接 JS-SDK、不做真实推送，也不保留双端设备切换；桌面、平板、手机宽度和企业微信内嵌浏览器访问同一套业务页面。

### 4.2 企业背景改为医药企业

原型 mock 数据不能是泛化 IT 项目，应体现医药企业内部 IT 项目场景。

代表项目包括：

- SAP、CRM 等内部业务系统。
- GxP 合规文档管理、LIMS 实验室系统等医药合规和研发相关系统。
- 算力中心搭建，例如 GPU 集群、研发 AI 建模算力平台。
- 运营指标报表、权限审计、培训中心音视频设备更新等内部管理和硬件集成项目。
- 外部供应商也应是 IT 供应商、实施商、顾问、验证咨询或硬件/机房实施供应商。

### 4.3 项目实现方式不是二选一

一个项目可能同时包含多种投入来源：

- 内部 IT 开发投入。
- 外部供应商投入。
- 业务方参与投入。

原型需要体现“合作实现分工”，不能只表达“内部实现”或“外部供应商”二选一。

当前 mock 数据同时保留三类实现方式，便于演示不同治理口径：

- 纯内部实现：例如运营指标自助报表门户，由 IT部完成产品方案、开发、接口和审计日志。
- 纯外部供应商实施：例如培训中心音视频设备更新，由外部集成商完整实施，IT部项目经理负责合同、交付和风险治理。
- 合作实现：例如 SAP、CRM、GxP、LIMS、算力中心，由内部 IT、外部供应商和业务部门共同参与。

### 4.4 资源与预算必须支持人员占用穿透

资源与预算视图不能只看一个人被分配了多少时间，还要能穿透查看：

- 这个人被哪些项目占用。
- 每个项目中承担什么具体工作。
- 每项工作占用多少小时。
- 当前工作状态是什么。

### 4.5 主界面需要卡片/看板式展示

各模块主界面不能只有列表。对于适合展示的模块，应支持卡片式或看板式视图，使演示更直观。

当前已落实到：

- 需求管理：卡片/列表切换。
- 项目管理：卡片/列表/项目甘特切换。
- 资源与预算：人员资源池卡片/列表/资源甘特切换。
- 任务与工时：看板式任务流。

### 4.6 角色左侧栏目必须按权限显示

不同角色不能在左侧看到全部模块。侧边栏应只展示该角色可访问的模块。

当前角色可见模块见本文档第 9 节。

### 4.7 提供白底与黑底商务风选择

为方便给老板或管理层演示，原型顶部应提供全局风格切换：

- 白底专业版：保持当前办公后台风格，适合需求评审、产品讲解和日常管理演示。
- 黑底商务版：偏深色管理驾驶舱和商务汇报风格，适合管理层演示和对比选择。

主题切换应是全局能力，覆盖侧边栏、顶部栏、卡片、表格、抽屉、弹窗、标签、表单和图表等基础组件。

### 4.8 开发资源排期日历

资源排期需要支持日、周、月三种颗粒度，展示开发人员每天具体处理什么 task，以及该 task 属于哪个项目。

展示结构要求：

- 每个排期块上行展示 task。
- 每个排期块下行展示项目。
- 日视图用于查看某一天上午/下午/全天的具体安排。
- 周视图用于默认查看一周内每天的 task 和项目。
- 月视图用于查看整月负载分布和忙闲情况。

权限入口要求：

- 管理视角在“资源与预算”中查看人员排期。
- 开发本人不进入“资源与预算”，应在“任务与工时”中看到自己的排期。

### 4.8.1 项目甘特与资源甘特

新增两类甘特视角：

- 团队项目甘特：放在“项目管理”，按项目展示从项目治理、研发实施到运营验收上线的时间跨度，用于项目组合和团队交付节奏演示。
- IT内部资源甘特：放在“资源与预算”，按人员展示项目/工作占用、负载风险和空闲情况，用于资源协调和排期冲突演示。

两类甘特都在单套 H5 页面中展示，手机宽度下通过局部横向滚动查看完整时间轴。甘特条必须来源于现有 mock 数据链路，不能单独造一套脱离项目、任务、资源排期的数据。

### 4.9 角色切换与侧边栏交互

顶部角色切换按钮不应占用顶部主操作区，应迁移到左侧底部，命名为“切换角色（演示）”。

左侧导航需要支持折叠和展开：

- 展开时显示品牌、模块图标、模块名称、角色切换下拉和集成提示。
- 折叠时只保留图标入口，减少导航占用宽度。

### 4.10 0-6 阶段泳道流程

需求从提出到验收评分采用一条 0-6 主流程，不再保留独立“流程工作台”：

- 阶段0 草稿：需求方形成需求草稿，完成标志为发起需求评审。
- 阶段1 需求评审：产品经理评审需求，填写价值评分、迭代版本、资源方案、预算测算、预算依据和实现决策；AI 评分辅助决策。
- 阶段2 方案确认：需求方确认产品方案，完成标志为放弃需求或发起项目申请。
- 阶段3 项目启动：唯一项目经理判断资源情况，决定启动或退回方案确认。
- 阶段4 项目进行：项目经理指派多个开发或外部供应商；开发拆分子任务、估时、登记工时、汇报进度并完成任务。
- 阶段5 项目验收：产品经理验收项目，完成或退回项目进行。
- 阶段6 验收完成：需求方提交 1-5 分和评价，流程关闭只读。

当前稳定口径：

- 阶段图放在需求详情和项目详情中，项目详情必须能看到完整 0-6 阶段、当前负责人、交付物、当前节点、动作记录和留言；不再做“泳道标签 + 表格行”布局。
- 项目启动及启动之前没有实际交付，项目交付进度一律显示为 0；只有进入项目进行后才展示真实交付进度。
- 管理层只读查看全局指标、报表和项目详情，不作为流程工作方展示在泳道中。
- 阶段推进按钮只对当前角色有权操作的节点高亮，其他节点置灰。
- 所有阶段动作支持填写留言，留言进入操作记录和通知。
- 节点设置是演示配置能力，只允许对应角色看到自己有权设置的节点，不做真实流程引擎或后端持久化。

### 5.4 项目管理

已实现：

- 项目卡片/列表切换。
- 项目甘特视图：按当前主责团队分组展示项目时间轴、阶段、进度、下一里程碑和今日线。

- 阶段条、里程碑、资源分配、关联任务。

- 合作实现分工，展示内部 IT、外部供应商、业务方的责任、投入、成本和状态。
- 项目详情中展示 0-6 阶段泳道、工作方、当前节点、动作记录和留言，供有权角色查看和推进。
- 项目卡片、列表、甘特条、预算燃尽、风险项目和待办项均可点击进入项目详情。

### 5.6 资源与预算

已实现：

- 资源指标卡：整体负载、资源申请、外采合同、高负载人员。
- 人员资源池卡片/列表/资源甘特切换。

- 人员占用穿透到项目、具体工作、小时数和状态。

- 资源申请列表。
- 转项目资源安排：展示资源申请节点下项目经理安排的具体人员、时间窗口、小时数和负载/冲突。
- 供应商预算列表。
- 项目预算展示需求评审阶段产品经理填写的预算测算，不允许项目页凭空生成预算。

### 5.7 绩效与报表

已实现：

- 按角色展示不同绩效域：高管/管理员可切换全局、项目、产品、开发、业务部门、供应商、预算；普通角色只看本人或本部门允许范围。
- 时间范围筛选：近30天、近3月、近6月、本年。
- 维度筛选：部门、项目类型、实现方式、阶段、风险、项目经理、产品负责人、供应商负责人。
- 项目、产品、开发、业务部门、供应商、资源预算等绩效卡片、趋势、排行、明细和管理洞察。

### 5.8 权限管理

已实现：

- 用户列表。
- 权限矩阵。
- 按角色预览菜单/数据范围。
- 角色切换时展示该角色可见模块、可操作动作和数据范围说明。
- 角色与部门分离展示：用户先属于组织部门，再通过角色获得菜单和数据范围。
- 部门负责人作为组织身份展示，不作为独立角色。
- 管理员为无部门全局账号；项目经理、产品经理和开发归属 IT部；需求方归属业务部门；管理层独立成部门。
- 唯一项目经理同时是 IT负责人；需求方负责人是业务部门负责人；管理层负责人保持为管理层组织身份。

说明：权限管理属于系统设置，仅管理员可见；当前只做原型展示，不做真实鉴权。

### 5.9 系统设置

已实现：

- 左侧“系统设置”为可展开栏目，仅管理员可见。
- 系统设置下包含：集成中心、权限管理、用户设置、角色设置、部门设置、通知设置。

## 11. 后续工作规则

后续继续制作原型时，必须遵守以下规则：

1. 先读本文档，确认原始目标、追加需求和当前实现范围。
2. 再读《IT项目管理系统原型演进故事.md》，确认关键演进口径。
3. 如果用户提出新需求，先判断是“补充需求口径”还是“具体实现任务”。
4. 如果是补充需求口径，更新本文档第 4 节或相关章节。
5. 如果是具体实现任务，只有在改变稳定口径、角色模型、组织模型或流程主线时才更新演进故事。
6. 如果新增模块、调整角色权限、替换 mock 数据主线或改变不做范围，必须同步更新本文档。
7. 完成代码改动后至少运行 `npm run build`；若只是文档改动，至少运行 `git diff --check`。
8. 新增 mock 时必须保证需求、项目、流程、任务、排期能通过 ID 互相印证，不能只写孤立展示文本。
9. 维护单套 H5 响应式页面：新增重要业务动作时，桌面和手机宽度都要可访问、无重叠，宽内容只能在自身容器内横向滚动。
