# 通知与订阅体系设计

最后更新：2026-05-26

## 1. 文档目的

本文档专门说明 IT 项目管理系统原型中的通知与订阅体系设计。

它回答三个问题：

- 系统里哪些事情会产生通知？
- 通知优先级、渠道和可配置性怎么定义？
- 为什么不同角色会看到不同通知和订阅项？

本文档是“消息通知”页面、“系统设置 / 通知设置”和“集成中心”通知渠道展示的设计依据。后续新增通知事件时，应先更新本文档，再更新 `src/data.ts` 中的 `notificationCatalog` 和 `roleNotificationSubscriptions`。

## 2. 设计原则

通知体系不能从角色开始拍脑袋配置。正确顺序是：

1. 先定义全局通知目录。
2. 再定义每个通知事件的优先级、渠道和可配置性。
3. 再按角色配置默认订阅、可选订阅和强制通知。
4. 最后才在页面上展示当前角色收到什么、能订阅什么。

这样做的原因是：如果没有全局目录，后续新增“供应商付款节点异常”“人员排期冲突”“MCP 调用失败”等事件时，就没有统一规则判断它属于哪个业务域、谁应该收到、是否能关闭、走什么渠道。

## 3. 核心概念

### 3.1 全局通知目录

全局通知目录是系统级的通知事件字典。

它不是用户收件箱，而是定义“系统里哪些事件会触发通知”的配置源头。

每条目录项包含：

- 通知域：事件来自哪个业务领域。
- 事件名称：具体发生了什么。
- 事件说明：什么时候触发。
- 通知优先级：高 / 中 / 低。
- 触达渠道：站内信、企业微信、机器人。
- 是否可配置：用户能不能关闭或调整。

### 3.2 通知域

当前通知域包括：

| 通知域 | 说明 |
| --- | --- |
| 需求 | 需求提交、优先级调整、进入验收等 |
| 项目 | 阶段变化、风险升级、里程碑延期等 |
| 任务 | 新任务分配、截止提醒、工时提醒等 |
| 资源 | 资源申请、人员超负荷等 |
| 预算 | 预算超阈值、外采金额异常等 |
| 验收 | 验收评分、评分异常等 |
| 权限 | 角色、用户、数据范围变化等 |
| 集成 | MCP 或企业微信机器人调用异常等 |

### 3.3 通知优先级

通知优先级不是需求优先级，也不是项目优先级。

它表示通知的打扰强度和处理紧急度。

| 当前命名 | 业务含义 | 建议展示解释 | 典型渠道 | 示例 |
| --- | --- | --- | --- | --- |
| 高 | 需要及时处理，影响交付、预算、风险或合规 | 强提醒 | 站内信 + 企业微信，部分不可关闭 | 高风险项目、预算超阈值、任务逾期、评分异常 |
| 中 | 需要关注，但不一定马上处理 | 关注提醒 | 站内信为主，必要时企业微信 | 阶段变更、验收评分提交、权限变更 |
| 低 | 仅记录或弱提醒 | 记录通知 | 站内信 | 普通同步、信息归档 |

后续如果觉得“高 / 中 / 低”不够业务化，页面展示可以改成“强提醒 / 关注提醒 / 记录通知”，但数据层仍可保留优先级字段。

### 3.4 通知渠道

| 渠道 | 作用 |
| --- | --- |
| 站内信 | 全量沉淀，作为系统内记录和追溯入口 |
| 企业微信 | 高优先级、待处理、临期、异常类提醒 |
| 机器人 | 查询、快捷更新、MCP 异常或对话式交互相关提醒 |

渠道不是所有通知统一打开。高优先级事件更可能触达企业微信，低优先级事件通常只保留站内信。

### 3.5 可配置性

通知分为三类：

| 类型 | 说明 |
| --- | --- |
| 默认订阅 | 系统按角色默认打开，用户可理解为角色职责的一部分 |
| 可选订阅 | 用户可以按关注点打开或关闭 |
| 强制通知 | 风险、预算、任务截止、合规等关键事件，不建议关闭或不可关闭 |

## 5. 角色订阅策略

### 5.1 高管

关注重点：管理例外、预算风险、重大交付风险、低评分异常。

默认订阅：

- 高风险项目预警。
- 关键里程碑延期。
- 预算超阈值。
- 评分异常。

可选订阅：

- 项目阶段变更。
- 验收评分提交。
- MCP 调用异常。

不默认订阅：

- 普通任务分配。
- 工时填报提醒。
- 普通需求状态流。

### 5.2 项目经理

关注重点：资源、进度、风险、预算、权限和跨项目治理。

默认订阅：

- 项目阶段变更。
- 高风险项目预警。
- 关键里程碑延期。
- 资源申请待审批。
- 人员超负荷。
- 预算超阈值。
- 任务截止提醒。
- 权限变更。

可选订阅：

- 新需求提交。
- 验收评分提交。
- MCP 调用异常。

### 5.3 产品经理

关注重点：需求承接、资源申请、实现进度和验收结果。

默认订阅：

- 新需求提交。
- 需求进入验收。
- 项目阶段变更。
- 资源申请待审批。
- 验收评分提交。

可选订阅：

- 需求优先级调整。
- 高风险项目预警。
- MCP 调用异常。

### 5.4 运营 + 运营中心负责人身份

关注重点：用户的业务角色仍是运营；当该用户同时是运营中心负责人时，额外关注本部门需求优先级、延期、验收和投入异常。

默认订阅：

- 需求优先级调整。
- 需求进入验收。
- 关键里程碑延期。
- 人员超负荷。
- 评分异常。

可选订阅：

- 新需求提交。
- 项目阶段变更。
- 验收评分提交。

### 5.5 开发

关注重点：个人执行任务、排期变更和工时提醒。

默认订阅：

- 新任务分配。
- 任务截止提醒。
- 工时填报提醒。

可选订阅：

- 项目阶段变更。
- MCP 调用异常。

不默认订阅：

- 预算超阈值。
- 供应商相关管理风险。
- 团队级资源投入异常。

## 6. 页面呈现规则

### 6.1 消息通知页

消息通知页是个人收件箱，不是全局规则维护入口。

消息通知页展示：

- 顶部摘要区：显示当前角色未读数量，提供通知设置入口和“全部已读”操作。
- 通知列表：展示当前角色可见的通知。
- 我的订阅摘要：展示当前角色会接收哪些类型的通知。

普通角色不能管理全局通知目录。管理员可通过顶部设置入口或左侧系统设置进入“通知设置”。

### 6.2 系统设置 / 通知设置

通知设置页只对管理员开放，负责维护系统级规则：

- 全局通知目录：展示系统全部通知事件，作为订阅规则的源头。
- 角色订阅策略：展示每个角色的默认订阅、可选订阅和强制通知。
- 通知渠道规则：展示站内信、企业微信、机器人等触达方式。

### 6.3 集成中心

集成中心只展示与渠道有关的能力：

- 哪些通知会进入企业微信。
- 哪些通知可通过机器人或 MCP 触达。
- 是否按角色订阅或强提醒。

集成中心不维护订阅策略，只引用全局通知目录。

## 7. 当前原型限制

当前仍是静态前端原型：

- 不做真实消息推送。
- 不接真实企业微信。
- 不接真实 MCP。
- 不保存订阅开关。
- 不做真实用户级个性化订阅。

当前重点是展示通知体系的产品逻辑，而不是实现真实通知服务。

## 8. 后续扩展方向

后续如果继续做真实产品设计，可以扩展：

- 用户级订阅偏好。
- 项目级订阅偏好。
- 企业微信免打扰时段。
- 通知升级规则，例如站内信未读 2 小时后升级企业微信。
- 通知去重和合并。
- 通知审计日志。
- MCP/机器人快捷处理入口。
