# IT项目管理系统原型演进故事

最后更新：2026-05-26

> 本文件记录“为什么原型变成现在这样”。它不是逐项施工日志，也不重复总览文档中的完整功能清单。继续开发前先读《IT项目管理系统原型总览与工作记录.md》，再读本文档最近的故事节点。

## 记录原则

本文件只保留会影响后续判断的上下文：

- 用户明确纠正过的业务口径。
- 原型方向发生变化的关键节点。
- 角色、组织、权限、数据范围、行业背景等稳定设定。
- 容易在后续对话压缩中遗失的设计原因。

不再记录：

- 每次改了哪些文件。
- 每次影响哪些页面的流水账。
- 每次 `npm run build`、`git diff --check` 的通过记录。
- 已经在总览文档、通知设计文档或代码结构中清楚体现的细节。

如果只是普通 UI 微调或 mock 数据补充，优先更新总览文档或对应设计文档；只有当它改变“故事走向”时才写进这里。

## 当前故事线

### 1. 从 PRD 到 PC 静态原型

原始 PRD 要求的是 IT 项目管理系统，覆盖需求、项目、任务、资源预算、绩效报表、权限、通知、MCP 和企业微信能力。

用户先明确：先做网页端原型，不做独立移动端闭环，不接后端、不接数据库、不做真实鉴权、不调用真实企业微信或 MCP。当前原型因此始终是“可演示的静态前端”，重点是把流程、角色视角和业务口径讲清楚。

### 2. 行业背景从泛化 IT 转为医药企业 IT

用户指出这是医药企业，不是泛化互联网 IT 管理系统。mock 数据因此切换到医药企业内部 IT 项目：

- SAP、CRM 等内部业务系统。
- GxP、LIMS 等合规和研发相关系统。
- 算力中心、GPU 集群、硬件和软硬件协同项目。
- 外部 IT 供应商、实施商、验证咨询、硬件和机房供应商。

这个阶段确立了一个长期口径：项目不能只按“软件项目”理解，还要覆盖硬件项目、软硬件协同；同时把实现方式单独拆出来，区分内部实现、外部供应商和合作实现。

### 3. 实现方式从二选一变为多方协作

用户强调：一个项目可能同时有内部开发投入、供应商投入和业务方投入，不是“内部 or 外部”的二选一。

因此原型加入了三类样例：

- 纯内部实现：如运营指标自助报表门户。
- 纯外部供应商实施：如培训中心音视频设备更新。
- 合作实现：如 SAP、CRM、GxP、LIMS、算力中心。

后续新增项目 mock 时，要继续保证需求、项目、任务、资源、预算、供应商、通知之间能联动，不要只在单个页面孤立出现。

### 4. 组织模型逐步收敛

用户多次纠正部门和人员归属，最终稳定为：

- `IT部`：项目经理、IT负责人，负责项目治理、资源协调、预算、风险、供应商交付管理。
- `研发部`：产品经理、开发、研发负责人，负责产品方案、内部开发、技术联调和交付支持。
- `运营中心`：运营、运营中心负责人，负责业务需求提出、确认、验收和部门投入视角。
- `管理层`：高管。
- `无部门全局管理员`：系统配置账号。

外部供应商不是内部部门，不进入部门设置。供应商由项目经理治理交付、合同、付款、预算和风险；产品经理参与选型、方案适配和验收组织。

旧口径如项目管理中心、质量管理部、市场部、数字化产品部等不再作为组织部门出现。

### 5. 角色和组织身份被拆开

一开始原型把“运营中心负责人”做成了独立角色。用户指出这是错的：负责人不是角色，而是组织身份。

当前稳定口径：

- 业务角色：管理员、高管、项目经理、产品经理、运营、开发。
- 组织身份：IT负责人、研发负责人、运营中心负责人等。

同一个人可以是某个业务角色，同时叠加部门负责人身份：

- 周宁：运营 + 运营中心负责人。
- 李书航：项目经理 + IT负责人。
- 许知远：开发 + 研发负责人。

负责人身份影响数据范围和可操作能力：

- 普通运营只看本人需求。
- 运营中心负责人看本部门需求，可介入修改、调整优先级、看部门投入。
- 普通开发看本人任务和排期。
- 研发负责人看研发部任务、多人排期、负载和技术联调。
- 普通项目经理看项目治理视角。
- IT负责人看 IT部项目组合、资源预算、供应商风险和绩效。

左下角“切换角色（演示）”实际切的是演示身份，不等同于真实系统里的角色表。

### 6. 权限管理归入系统设置

用户指出：权限管理不应该挂在项目经理下面。项目经理可以看项目参与人或项目治理信息，但不能管理用户、角色、菜单和全局权限。

当前稳定口径：

- `权限管理` 属于 `系统设置` 子项。
- 只有管理员能进入系统设置和权限管理。
- 项目经理不再看到权限管理入口。
- 用户设置展示角色和组织身份。
- 角色设置只展示业务角色，不把部门负责人身份当作角色。

这个口径后续不能再倒回“项目经理管理全局权限”。

### 7. 需求方命名为运营

用户不接受“需求方”这个名字，要求改为“运营”。因此原型展示口径统一为：

- 普通业务提交人叫“运营”。
- “需求方”可以只作为 PRD 原始概念或历史说明，不应作为当前界面的角色名。

普通运营的主入口是需求管理；消息通知通过右上角个人入口进入。

### 8. 需求转项目曾经扩展为跨角色泳道流程

用户提出：需求转项目不是简单状态条，需要像泳道图一样展示流程节点，并能设置节点。

当时的流程节点为：

`需求提出 → 产品经理评估 → 业务方确认 → 项目资源申请 → 迭代计划 → 开发中 → 验收中 → 上线`

泳道包括：

`运营中心`、`产品经理`、`项目经理 / IT部`、`研发部 / 开发`、`外部供应商`、`高管关注`

关键设计原因：

- 需求管理展示需求视角。
- 产品工作台作为主入口，展示产品经理如何推进需求转项目。
- 项目管理详情展示转项目后的交付视角。
- 资源与预算承接“项目资源申请”节点，展示项目经理安排哪些人、哪些时间。

后续第 13 节已经把它升级为“主业务流程 + 产品侧状态 + 项目交付侧状态”。因此泳道图现在只作为协作链路和交接记录，不再作为唯一主流程。节点设置只是演示能力，不做真实流程引擎、拖拽编排、审批流或后端持久化。

### 9. 资源与预算必须能穿透到人

用户强调资源与预算不能只看总时间，要能穿透到“这个人到底被什么项目和工作占据了时间”。

因此资源与预算形成了两个层级：

- 管理视角：人员资源池、项目投入拆解、供应商预算、资源申请。
- 人员穿透：点击人员后看项目占用、具体工作、小时数、状态和日/周/月排期。

开发本人不进入资源与预算，而是在任务与工时里看自己的日/周/月排期。研发负责人因为有部门负责人身份，可以看研发部任务和多人排期。

### 10. 通知体系从个人消息拆出全局配置

用户指出：不能拍脑袋给不同角色通知，必须先有顶层通知目录和订阅规则。

当前稳定口径：

- 个人资料里只有个人消息通知。
- 全局通知目录、角色订阅策略、渠道规则归入 `系统设置 / 通知设置`。
- 只有管理员可以管理全局通知设置。

通知设计细节以《通知与订阅体系设计.md》为准。

### 11. 导航和演示壳调整

为了适合 PC 原型演示，用户要求：

- 左侧菜单按当前身份权限展示，不能所有角色都看到全部模块。
- 左侧栏可折叠，折叠后只保留图标。
- 角色切换从顶部挪到左下角，命名为“切换角色（演示）”。
- 顶部右侧放 light/dark、通知铃铛和个人资料入口。
- 消息通知在个人资料页中，与个人信息并列为 tab。
- 右侧内容滚动不应拉长左侧边栏，左右独立滚动。

这些是演示体验层面的稳定设定。

### 12. 产品经理和项目经理必须能推进业务

用户指出产品经理、项目经理不能只是看流程和报表。原始 PRD 中，产品经理负责承接需求、分析、申请资源、制定迭代和组织验收；项目经理负责资源、预算、进度、风险和供应商治理。

因此泳道图的稳定口径调整为：

- “节点设置”是流程配置能力，不代表真实推进。
- “业务动作”才是推进流程的主入口。
- IT负责人可以指定某个产品经理承接需求，作为产品承接环节的开始。
- 产品经理通过确认承接、保存产品评估、提交项目申请、发起验收推进业务。产品评估同时包含范围、实现方式、资源测算、供应商适配和验收标准。
- 项目经理通过受理项目申请、转项目、确认资源排期、更新风险应对、更新预算和供应商交付、进入联调/验收支持、提交上线归档推进项目。
- 部门负责人分配工作是业务开始的信号，分配后要生成具体到人的消息通知。
- 每次推进都要留下流程或项目操作记录，并同步生成通知。

通知也随之从静态消息扩展为业务事件，包括工作分配、产品承接、产品评估确认、项目申请提交/退回/受理、资源排期确认、项目记录修改、交付阶段推进、风险应对更新和验收发起。

这一阶段之后继续细化：产品经理和项目经理不能共用一套含糊的“推进阶段”。产品经理负责把需求变成可交付方案，项目经理负责把方案变成受治理的交付项目。

### 13. 主业务流程与产品 / 项目分工

后续原型以这条主业务流程为准：

`需求提交 → 业务确认与优先级 → 产品指派与承接启动 → 产品评估 → 项目申请 → 项目受理与立项 → 资源排期与任务拆解 → 实施与联调 → 业务验收 → 上线归档`

这是系统的主干，不再把一个“需求转项目泳道图”当成唯一事实来源。泳道图可以继续展示跨角色协作链路和交接记录，但真正的状态要分成两套：

- 需求 / 产品侧状态：`待业务确认 → 待产品承接 → 产品评估中 → 待项目受理 → 交付中 → 待验收 → 已完成`
- 项目 / 交付侧状态：`待受理 → 已立项 → 资源排期中 → 实施中 → 联调测试中 → 验收支持中 → 已上线 → 已归档`

角色边界固定如下：

- 运营：提交需求、补充业务信息、参与业务确认和验收评分。
- 运营负责人：确认本部门需求、调整优先级、确认范围和验收结论。
- IT负责人：在产品承接前指定产品经理，必要时复核重大资源、预算和风险。
- 产品经理：确认承接、做产品评估、在同一评估里确定范围 / 实现方式 / 资源测算 / 验收标准、提交项目申请、组织业务验收。
- 项目经理：受理项目申请、立项、排资源、治理预算 / 风险 / 供应商、推进交付阶段、上线归档。
- 开发：承接任务、更新状态、填报工时、完成联调和技术问题处理。

审批和通知只在关键交接点出现：运营提交需求、运营负责人确认、IT负责人指派产品经理、产品承接、产品评估确认、项目申请提交/退回/受理、资源排期确认、任务分配、联调完成、验收发起/退回、上线归档、高风险或预算异常。

对应页面口径：

- 产品工作台是产品经理推进“需求到方案”的工作台，也给 IT负责人提供产品指派入口。主要区域是产品指派与承接启动、产品评估、项目申请、验收组织和退回补充。
- 项目管理是项目经理推进“方案到交付”的工作台，主要区域是项目申请池、项目受理与立项、资源排期、执行跟踪、风险预算供应商治理和上线归档。
- 项目申请是产品工作台和项目管理之间的正式交接物，必须和需求、项目、资源排期、通知互相印证。

### 14. Mock 数据必须能互相印证

用户指出项目详情里的“关联任务”不能只是文字列表。项目、任务、排期必须是同一套 mock 数据链路：

`Demand.id → Project.demandId → Flow.projectId / demandId → Project.taskIds → Task.projectId → ResourceCalendarEntry.taskId / projectId`

因此项目详情中的关联任务改为真实任务记录，能看到任务编号、负责人、状态、截止时间和工时。具备任务权限的角色可以从项目详情跳到任务与工时，并自动带上项目和任务筛选；没有任务权限的角色只看摘要，不出现跳转按钮。

### 15. 单套 H5 响应式原型

用户确认不再维护独立移动端预览模式，要求删除手机壳、底部 Tab 和移动端专用页面，改为单套 H5 响应式原型。V1.3 中的企业微信移动办公能力在本原型中统一解释为“企业微信内嵌 H5 自适应访问”。稳定口径如下：

- 不另起真实小程序工程，不保留双端设备切换。
- 所有角色、模块、通知和业务操作继续使用同一批 mock 数据和前端内存状态。
- 桌面宽度保持后台管理布局；手机宽度下同一套页面改为顶部横向导航、单列内容和局部横向滚动。
- 不新增手机专用底部导航或移动专属页面；新增业务能力必须在同一套 H5 页面中完成响应式适配。

### 16. 清爽化信息架构与详情页

V1.3 对齐时，用户明确提出页面信息密度过高、项目和需求详细内容不要放侧边栏、字体和样式需要优化。稳定设计如下：

- 需求管理和项目管理列表页只展示判断所需的摘要字段，长描述、流程、日志、资源分工和操作记录进入详情页。
- 需求详情和项目详情使用应用内页面状态跳转，不引入真实 URL 路由，不再用右侧抽屉承载需求/项目详情。
- 项目列表、项目申请池和项目详情显性展示项目立项 AI 评分，用 mock 数据展示业务价值、紧急程度、可行性、加权总分、权重规则、推荐结论和理由。
- 样式保持 Windows/macOS 通用字体栈，减少过多颜色、阴影和英文装饰，整体转向清爽企业 H5 风格。

### 17. 甘特图分成项目组合和资源投入两条线

用户要求“每个团队都有 by 项目的甘特图视角”，同时 IT 内部资源也要有甘特图。稳定设计如下：

- `项目管理 / 项目甘特` 回答“每个团队或当前主责团队的项目在什么时间推进到什么阶段”。行是项目，分组是 IT部项目治理、研发部交付实施、运营中心验收上线。
- `资源与预算 / 资源甘特` 回答“内部人员在什么时间被哪些项目和工作占用”。行是人员，条形来自人员占用和资源排期。
- 单套 H5 页面使用横向时间轴、月份刻度、今日红线、彩色条形、图例和风险提示；手机宽度下甘特图在自身容器内横向滚动，不造成整页横向溢出。
- 甘特数据不能孤立 mock。项目甘特从 `Project / Task / Milestone` 派生；资源甘特从 `ResourcePerson / ResourceCalendarEntry / Project` 派生。出现项目名或任务 ID 对不上时，应暴露为数据异常，不允许为了好看硬画孤立条。
- 后续补充 mock 数据时，必须继续保证 `Demand → Project → Flow → Task → ResourceCalendarEntry` 能互相印证。

### 18. 组织和主流程按 0-6 阶段重构

用户重新定义了当前原型的主流程，后续以此为最新口径，早期“运营中心 / 研发部 / 双状态流”的描述只作为历史记录。

组织边界调整为：

- `业务部门`：需求方、需求方负责人。需求方负责人可以查看全部需求-项目情况，并修改需求重要级别。
- `IT部`：唯一项目经理 / IT负责人、产品经理、开发。项目经理有且只有一个，同时是 IT部门负责人。
- `管理层`：管理层和管理层负责人，只做全局只读视角，不进入流程泳道占位。
- 删除研发部；运营部门统一改为业务部门。

主流程调整为：

`需求评审 → 方案确认` 与 `项目准备 → 项目启动 → 项目进行 → 项目完成 → 项目结束`

角色动作边界如下：

- 需求评审：产品经理评审需求，AI 评分辅助决策，并填写方案、资源、预算和实现决策。
- 方案确认：需求方确认方案。确认后需求等待产品经理预创建项目。
- 项目准备：产品经理创建项目并填写内部人天、外部供应商需求和外采人天。
- 项目启动：项目经理填入开发和外部供应商资源后进入启动。
- 项目进行：产品经理填写启动结束时间，开发和供应商拆分任务、估时排期、登记工时、汇报进度和完成任务。
- 项目完成：DDL 到期或产品经理点击后进入，等待需求方验收评分。
- 项目结束：需求方提交评分和评价，项目和关联需求关闭。

所有阶段推进动作都必须支持留言，留言进入操作记录和通知。页面上的节点设置只展示当前角色有权设置的节点；当前可操作节点高亮，其他节点置灰。

交付进度只代表项目进入实施后的真实交付完成度。项目启动及启动之前只能说明评审、方案和资源判断进展，不能计入交付进度，因此项目列表、项目详情、甘特、驾驶舱和报表中该阶段一律显示 0。

### 19. 流程工作台下线，详情页承载流程

独立“流程工作台”会和工作台、需求详情、项目详情、项目管理、任务与工时产生重复，因此下线。

新的承载方式：

- 工作台只做角色摘要、风险、待办和可点击入口。
- 需求详情承载需求方提交内容、产品经理需求评审填写区、当前阶段动作、留言和操作记录。
- 项目详情承载完整 0-6 阶段图、项目启动/指派/验收动作、资源与供应商安排、任务、风险、预算和操作记录。阶段图不再做表格泳道，改为一行阶段标题、一行阶段卡片，卡片直接展示当前负责人和交付物。
- 任务与工时承载开发的排期日历、按阶段查看、任务列表和任务卡片；视图切换只保留一处，不重复展示。

后续新增流程能力时，不再恢复独立流程工作台，应优先落到需求详情或项目详情。

### 20. 预算必须来自产品经理需求评审

用户指出项目里的预算不能像凭空出现。稳定口径改为：

- 预算由产品经理在阶段1“需求评审”填写，包括 `budgetEstimate` 和 `budgetBasis`。
- 产品经理保存需求评审后，关联项目通过 `Demand.id -> Project.demandId` 同步预算金额。
- 项目列表、项目详情、资源与预算、预算燃尽都必须显示预算来源为“需求评审”。
- 需求内容区不再展示产品经理填写的价值评分、迭代版本、资源方案和实现决策；这些字段属于阶段1产品经理评审填写区。

这条链路的目的是让费用、资源和项目治理有来源可追溯，而不是在项目页单独维护一套孤立预算。

### 21. H5、驾驶舱和项目详情收口

后续连续迭代把原型从“PC 后台”收口为“单套 H5 业务原型”。稳定口径如下：

- 不再维护独立企业微信小程序工程。当前原型是一套响应式 H5：桌面使用左侧导航，手机宽度和企业微信内嵌浏览器使用顶部横向导航、单列内容和局部横向滚动。
- 高管工作台迁移为 `IT项目驾驶舱`，默认深色商务风格，同时支持 light 主题。驾驶舱按全公司、财务、采购、临床、人事、IT部切换视图；财务使用当前主 mock 数据，其他部门使用页面内自洽的部门级 mock。
- 项目甘特和资源甘特正式拆开：项目管理看项目组合排期，资源与预算看 IT 内部人员占用。甘特条必须能落到现有项目、任务或资源排期，不允许孤立绘制。
- 项目详情只保留真正承载业务判断的内容：项目标题、关键指标、需求线/项目阶段关系图、AI 立项建议、关联任务和操作记录。基础信息、依赖关系、里程碑、阶段计划与交付物等重复区块下线。
- 阶段 1 `项目准备` 不允许出现开发关联任务。这个阶段只有产品经理提交的内部资源需求、外部供应商需求和外采人天；项目经理在“项目经理业务动作”中点击 `分配开发 / 供应商`，打开分配页面，确认后才进入阶段 2 `项目启动`。
- 阶段 2 `项目启动` 仍不产生开发任务。只有进入阶段 3 `项目进行` 后，开发和供应商才可以拆分任务、估时、登记工时、生成关联任务和排期。
- 项目详情中的关联任务必须通过 `Project.taskIds -> Task.id -> ResourceCalendarEntry.taskId/projectId` 互相印证。阶段 1/2 项目不得挂任务 ID，也不得在任务列表或资源日历中出现提前占用。

这次收口的核心判断是：页面可以简化，但 mock 数据不能撒谎。阶段、动作、任务、排期和通知必须互相证明同一个业务事实。

## 后续维护方式

当出现新需求时，先判断它属于哪一类：

- 改变业务口径：更新总览文档和本文档。
- 新增/调整通知规则：更新通知设计文档。
- 新增普通功能或 UI：只在代码和总览中体现，本文档不做流水账。
- 修复小问题：不写故事，除非它反映出角色、组织、权限或流程口径变化。

本文档要保持短、像故事、有判断；不要再变回千行施工日志。
